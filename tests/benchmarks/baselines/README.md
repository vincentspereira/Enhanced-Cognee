# Benchmark Baselines

This directory holds reference benchmark runs of
`tests/benchmarks/run_provider_comparison.py` against the full
Enhanced Cognee stack. CI compares new runs against these baselines
to detect performance regressions.

## What's here

| File | Generated by | Purpose |
| --- | --- | --- |
| `regression_thresholds.json` | Hand-written | Per-metric allowed regression % (e.g. p95 must not grow > 20%) |
| `2026-05-21_neo4j_stack.json` | Real Locust run | Baseline for the `neo4j_stack` permutation captured 2026-05-21 against a live stack (Neo4j 5.25 + Qdrant 1.12 + Valkey 8 + PostgreSQL 18 + pgvector). 60s, 20 users, 5 spawn-rate. |
| `2026-05-21_neo4j_stack.md` | Real Locust run | Markdown summary of the same run (per-endpoint p50/p95/p99). |

> Other permutations (`default`, `lean`, `embedded`, `memgraph_kuzu`)
> are not yet baselined -- they need their respective stacks
> (ArcadeDB / Apache AGE / ladybug+lancedb / memgraph+kuzu) running
> before a baseline run can be captured. `compare_to_baseline.py`
> handles "no baseline" gracefully (treats every permutation as
> "[NEW] in new run but no baseline -- skipped"), so CI's regression
> gate is inert for those permutations until baselines land.

## Regenerating a baseline

```bash
# 1. Boot the stack
docker compose -f docker/docker-compose-enhanced-cognee.yml up -d

# 2. Boot the MCP HTTP variant
python -m src.enhanced_cognee_mcp &

# 3. Run the benchmark
python -m tests.benchmarks.run_provider_comparison \
  --duration 120s \
  --users 50 \
  --spawn-rate 10 \
  --output-dir tests/benchmarks/baselines/output

# 4. Move + rename the result
mv tests/benchmarks/baselines/output/provider_comparison.json \
   tests/benchmarks/baselines/$(date +%Y-%m-%d)_default_stack.json
mv tests/benchmarks/baselines/output/provider_comparison.md \
   tests/benchmarks/baselines/$(date +%Y-%m-%d)_default_stack.md
```

## Regression check (CI)

`tests/benchmarks/compare_to_baseline.py` reads a new run + the
matching baseline + `regression_thresholds.json` and exits non-zero
if any metric regressed beyond its threshold.

```bash
python -m tests.benchmarks.compare_to_baseline \
  --baseline tests/benchmarks/baselines/<YYYY-MM-DD>_default_stack.json \
  --new tests/benchmarks/baselines/output/provider_comparison.json \
  --thresholds tests/benchmarks/baselines/regression_thresholds.json
```

## Why no synthetic baseline ships

A synthetic baseline (representative numbers from spec, not a real
run) would mislead the regression check -- new runs against the
"baseline" would always look like a regression or improvement
depending on whether the synthetic numbers were optimistic or
pessimistic. Better to ship no baseline and have CI's regression
gate be inert until a real one lands, than to ship fake numbers.
The `compare_to_baseline.py` tool handles "no baseline" gracefully
(treats every permutation as "[NEW] in new run but no baseline --
skipped").

The single baseline that ships in this directory (`2026-05-21_neo4j_stack.json`)
comes from an actual Locust run against a live stack -- not a hand-crafted
estimate.
